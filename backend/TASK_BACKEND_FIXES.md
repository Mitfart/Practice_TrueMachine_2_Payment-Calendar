# Задание: привести backend к техническому заданию

Источник требований: `../Техническое_Задание.docx`, зеркало: `../docs/technical-specification.md`.
Контекст текущего состояния: `backend/CONTEXT.md`.

## Цель

Закрыть основные расхождения backend с ТЗ по платёжному календарю без лишней архитектуры и новых зависимостей.

## Обязательные исправления

### 1. Валидация заявок на платёж

Файл: `app/Http/Controllers/PaymentController.php`.

- `priority` должен принимать только значения из БД: `low`, `normal`, `high`.
- `status` должен принимать только значения: `draft`, `on_approval`, `approved`, `in_registry`, `paid`, `rejected`.
- Не принимать произвольные строки статуса/приоритета.

### 2. Фильтры календаря

Файл: `app/Http/Controllers/CalendarController.php`.

Добавить фильтры из ТЗ:

- `item_id`
- `counterparty_id`
- `status` для платежей

Текущий `account_id` оставить.

### 3. Сводная строка календаря

Файл: `app/Http/Controllers/CalendarController.php`.

Добавить в ответ календаря сводку по всем выбранным счетам за каждый день:

- общий приход
- общий расход
- общий остаток
- признак кассового разрыва, если общий остаток меньше 0

Не ломать текущий формат по счетам, если frontend уже его использует. Лучше добавить отдельный ключ `summary`.

### 4. Формирование реестра по приоритету и доступному остатку

Файл: `app/Http/Controllers/RegistryController.php`.

Сейчас реестр берёт все согласованные платежи на дату/счёт. Нужно:

- брать только `approved`
- сортировать по приоритету: `high`, `normal`, `low`
- учитывать доступный остаток на дату
- добавлять платёж в реестр только если после него остаток не станет отрицательным
- если ни один платёж не прошёл — вернуть понятную ошибку

Деньги считать только в копейках, без float.

### 5. Частичный аудит изменений заявки

Минимальный вариант без новой большой системы аудита:

- при согласовании уже есть `approvals` — оставить
- для изменения даты/суммы/статуса заявки добавить простую запись истории, если таблица уже есть
- если таблицы нет — создать минимальную `payment_changes`: `payment_id`, `user_id`, `field`, `old_value`, `new_value`, timestamps

Не делать универсальный audit framework.

## Необязательные, но требуются по ТЗ для полной сдачи

- Повторяющиеся платежи по расписанию.
- Импорт планируемых движений из файла.
- PostgreSQL + Docker Compose запуск одной командой.
- OpenAPI-документация REST API.
- Экспорт всех отчётов, не только остатков/реестра.
- Админское управление пользователями.

## Проверка

Минимум после исправлений:

```bash
cd backend
php artisan test
```

Если тестов нет или окружение не готово — проверить хотя бы:

```bash
cd backend
php artisan route:list
php artisan migrate:fresh --seed
```

## Ограничения

- Не добавлять зависимости без необходимости.
- Следовать Laravel conventions.
- Не использовать float для денег.
- Не распылять расчёт остатков по разным местам, если появится общий метод — использовать его.
